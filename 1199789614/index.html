Go 语言网络编程概念

发表于 2024-06-24 更新于 2024-07-23 分类于 Go ， 5.网络编程 阅读次数： 本文字数： 3.5k 阅读时长 ≈ 12 分钟

HTTP 协议

HTTP（超文本传输协议，Hypertext Transfer Protocol）是一种用于分布式、协作式和超媒体信息系统的应用层协议。HTTP 是整个互联网基础，自 1991 年首次发布以来，经历了 3 次迭代修改，目前广泛使用的版本是 HTTP/1.1 和 HTTP/2。

HTTP 每个新版本都包含大量新特性，但是核心特点没有太多变化：

  • 无状态：HTTP 是一个无状态协议，服务器不会在两个请求之间保留任何数据（状态）。虽然协议简单可靠，但也限制了应用进行连续交互的能力。为此，通常在应用层使用诸如 Cookies、Tokens 之类的机制来存储状态信息。相对的，类似 FTP、SSH、Telnet 这类面向连接的协议，会在客户端和服务器之间创建一条持续通道。

  • 客户端-服务器结构：在 HTTP 中，客户端（user-agent，用户代理）发起请求，服务器等待请求并发送响应。这种请求-响应模式简化了通信流程，让客户端和服务器的职责明确。

  • 可扩展性：HTTP 头允许传输各种类型的数据，同时也支持内容协商。除了传输 HTML 文档，还可以传输任何类型的数据，包括图像、视频、JSON、XML 等。

  • 多种方法：HTTP 定义了一系列的方法，如 GET、POST、PUT 等，使得客户端能够执行不同类型的动作。

HTTP/2

HTTP/2 主要改进体现在网络性能优化上，虽然兼容 HTTP/1.1 语义，但和 HTTP/1.1 比在许多关键方面有显著区别：

  • 二进制协议：HTTP/1.1 基于文本，容易阅读和理解，但效率不高。HTTP/2 是基于二进制的，使用二进制帧来传输数据，使得协议的解析、扩展和优化更加高效和简洁。
  • 多路复用：HTTP/1.1 在同一时间内，每个 TCP 连接只能处理一个请求-响应，要发起多个请求必须建立多个 TCP 连接，容易导致网络阻塞。HTTP/2 支持多路复用，允许多个请求在单一 TCP 连接上同时进行，这样允许多个请求和响应交错传输，减少了阻塞问题。
  • 压缩头部：HTTP/1.1 每次请求和响应都必须重复发送 HTTP 头。HTTP/2 使用 HPACK 压缩算法压缩请求和响应的头部信息，减少了重复头部数据的传输。
  • 服务器推送：HTTP/1.1 只支持由客户端发起请求。HTTP/2 允许由服务器推送，预先发送需要资源到客户端。

HTTP 请求

HTTP 请求由一系列文本行组成，分别为请求行（request-line）、零到多个请求头部（header）、可选的请求体：

POST /api/data HTTP/1.1
Host: example.com
Content-Type: application/json
Content-Length: 81

{
    "name": "John Waters",
    "email": "jw@example.com",
    "age": 90
}

第一行为请求行，包括请求方法（request method）、URI（Uniform Resource Identifier，统一资源标识符） 地址和 HTTP 版本，后面三行为请求头，以空行结束。最后部分为请求体。

请求方法

HTTP 请求行的首要组成部分是请求方法，定义了对特定资源执行的操作类型。常见的方法包括：

  • GET：请求获取指定资源。
  • HEAD：与 GET 类似，但服务器在响应中只返回头部，不返回实际的数据，用于获取响应头部。
  • POST：用于提交数据，例如表单提交。
  • PUT：用于替换目标资源。
  • DELETE：请求删除指定的资源。
  • PATCH：用于对资源进行部分修改。

其中，在 HTTP/1.1 版本规定必须实现的方法只有 GET 和 HEAD，其他方法可选，也可自行定义增加。

请求头部

请求头部记录了与请求本身以及客户端有关的信息。请求头由任意多个用冒号分隔的纯文本键值对组成，以换行和回车结尾。

常见的请求头如下：

请求头部 描述
Accept 告诉服务器客户端能够接收的内容类型
Accept-Charset 浏览器可以接受的字符编码集
Accept-Encoding 浏览器可以支持的压缩类型
Accept-Language 浏览器偏好的语言
Authorization 包含基本身份验证证书
Cache-Control 指定请求和响应遵循的缓存机制
Connection 控制是否需要持久连接
Content-Length 请求体的大小（单位为字节）
Content-Type 请求体的 MIME 类型，如 application/json, text/html 等
Cookie 服务器发送的 Cookie
Host 请求发送到的服务器的域名和端口号
Origin 发起一个针对跨域资源共享的请求（CORS 请求）的来源 URI
Referer 包含请求来源 URL
User-Agent 包含客户端应用程序的标识

大部分请求头是可选的，只有 Host 是强制要求的头部。带有可选头部时，必须带有长度（Content-Length）或编码（Transfer-Encoding）字段。非标准请求头字段以 X- 作为前缀。

请求体

HTTP 请求中的请求体是一个可选部分，包含发送给服务器的数据。请求体主要用于那些需要向服务器提交数据的方法，如 POST、PUT、PATCH 等，GET 和 HEAD 请求通常没有请求体。

请求体的内容可以是多种类型的数据，由请求头中的 Content-Type 指定。常见类型包括：

  • application/x-www-form-urlencoded：最常用的 POST 数据提交方式，表单提交的默认方式。发送前编码所有字符（空格转换为 + 加号，特殊符号转换为 ASCII HEX 值）。
  • multipart/form-data：当表单中需要上传文件时采用的格式。
  • application/json：用于发送 JSON 格式的数据。
  • text/xml 或 application/xml：用于发送 XML 数据。
  • application/octet-stream：通常用于传输文件或其他二进制数据。

HTTP 响应

HTTP 响应是由服务器发往客户端的消息，是对之前客户端发起的 HTTP 请求的回应。和 HTTP 请求一样，HTTP 响应也由一系列文本行组成，分别为状态行、零到多个响应头部、可选的响应体：

HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 1234
Server: Apache/2.4.1 (Unix)

<html>
<head>
    <title>Example</title>
</head>
<body>
    <h1>Hello, world!</h1>
</body>
</html>

第一行为状态行，包括 HTTP 版本、状态码（status code）和状态消息（reason phrase），后面三行为响应头，以空行结束。最后一部分为响应体。

响应状态码

响应状态码为一个三位数字，分为五大类，每一类表示不同类型的响应：

  • 1XX (Informational) 信息响应：表示临时响应，请求已被接收并且客户端应继续请求。

  • 2XX (Success) 成功响应：表示客户端请求已成功被服务器接收、理解并接受。

  • 3XX (Redirection) 重定向：表示客户端需要采取进一步的操作以完成请求，通常是重定向。

  • 4XX (Client Error) 客户端错误：表示客户端可能发生了错误，导致服务器无法处理请求。

  • 5XX (Server Error) 服务器错误：表示服务器在处理请求时发生了错误。

响应头部

由冒号分隔的键值对形式的文本，提供了有关服务器、响应体和请求本身的额外信息。

常见的响应头部：

响应头部 描述
Content-Type 指定响应体的类型，如 text/html、application/json 等
Content-Length 响应体的长度，以字节为单位
Content-Encoding 响应体使用的编码类型，如 gzip、deflate 等
Set-Cookie 服务器向客户端浏览器发送的 Cookie
Cache-Control 指示响应内容是否可以被缓存，以及缓存的条件
Expires 响应内容过期的日期和时间
Last-Modified 资源的最后修改日期和时间
ETag 资源的特定版本的标识符，通常用于缓存优化
Server 包含处理请求的服务器信息，如 Apache、Nginx
WWW-Authenticate 告知客户端进行认证的方法
Location 用于重定向，提供重定向的目标 URL
Strict-Transport-Security HTTP 严格传输安全，强制客户端使用 HTTPS 连接

响应体

HTTP 响应体是服务器返回给客户端的数据部分，可以是 HTML 文档、JSON 数据、二进制文件等，不是所有响应都有响应体。响应体的具体类型由 HTTP 响应头中的 Content-Type 指定：

  • text/html：标准的 HTML，通常用于展示网页内容。
  • application/json：JSON 格式的数据，常用于 API 响应。
  • text/plain：纯文本数据。
  • image/png、image/jpeg：图像文件。
  • application/xml：XML 数据。

响应体还可以通过 Content-Encoding 响应头进行编码，例如使用 gzip 压缩以减少传输数据的大小。

URL 地址

URL（Uniform Resource Locator，统一资源定位符）是用于定义互联网资源位置的地址，可以用来访问网页、图像、视频和其他文件。完整的 URL 定义：协议://用户名:密码@子域名.域名.顶级域名:端口/目录/文件名.文件后缀?参数=值#锚点，具体各部分解析如下：

  • 协议：也称为方案，定义了服务器和客户端之间如何交换信息。常见的协议包括 http、https、ftp 等
  • 主机名：通常是服务器的域名或者 IP 地址。
  • 端口（可选）：端口号用于访问服务器上的特定服务，如果省略了端口号，则使用协议的默认端口。
  • 路径：路径指向服务器上的特定资源。
  • 查询字符串（可选）：开始于问号 ?，其后是一个或多个键值对参数，参数之间用 & 符号分隔。查询字符串用于向服务器发送额外的信息，常用于表单提交或页面请求的详细信息。
  • 锚点（可选）：也称为片段标识符，始于井号 #，用于使浏览器定位到网页的特定部分。

Web 应用

Web 应用（Web Application）是通过网络为用户提供功能或服务的应用程序，常见的有网上商城、网上银行、社交媒体、协作平台和流媒体服务等。

典型的 Web 应用有几个特点：

  • 客户端-服务器架构：Web 应用通常采用客户端-服务器模型，客户端负责呈现用户界面和用户交互，而服务器则处理后台数据处理、业务逻辑和数据库交互。

  • 跨平台访问：用户可以通过任何支持现代网络浏览器的设备访问 Web 应用，支持多种操作系统和设备。

  • 集中更新和维护：开发者在服务器端集中更新应用，所有客户都能即时访问到应用的最新版本。

  • 交互性：现代 Web 应用通过动态网页技术，提供与传统桌面应用类似互动性。

  • 依赖于网络连接：大多数 Web 应用依赖于活跃的网络连接。

从 Go 语言开发角度，Web 应用也能被分为处理器（handler）和模板引擎（template engine）两部分，处理器负责处理业务逻辑和 HTTP 请求，而模板引擎则负责生成动态的 HTML 内容。

Socket

Socket 是网络通信过程的一个端点，提供了操作系统或网络服务的一个编程接口。Socket 使应用程序能够读写与网络协议栈的交互，通过 Socket 网络中运行的程序可以发送和接收数据包。

Socket 通常用于实现网络中的客户端和服务器之间的通信。一个服务器程序会创建一个 Socket，然后监听指定的网络地址和端口，等待客户端的连接请求。客户端程序也会创建一个 Socket，用来向服务器的指定地址和端口发起连接请求，一旦连接建立，客户端和服务器就可以通过 Socket 交换数据。

Socket 类型

在网络编程中常见的类型有三种：

  • 流式套接字（Stream Sockets）：也称为 TCP Socket（SOCK_STREAM），是一个可靠的、面向连接的、基于字节流的通信通道，保证数据正确性和顺序性。

  • 数据报套接字（Datagram Sockets）：也称为 UDP Socket（SOCK_DGRAM），是一个不可靠的、无连接的服务，通过发送独立的包进行通信。UDP Socket 不保证消息的顺序或可靠性，所以速度一般比 TCP Socket 快，但是必须由应用程序处理可能出现的错误和数据丢失。

  • 原始套接字（Raw Sockets）：允许对较低层次的协议（如 IP 或 ICMP）直接访问，用于需要细粒度控制数据传输的应用，常用于开发新的自定义协议。

RPC

RPC（Remote Procedure Call，远程过程调用）是一种在网络中的不同计算机之间通信的技术，允许一台计算机上的程序调用另一台计算机上的程序或方法，就像这些程序运行在同一台服务器中一样。RPC 抽象了网络通信的细节，让开发可以专注于业务逻辑的实现，而不用深入处理网络交互的复杂性。

RPC 的基本工作原理是将一个过程调用的参数打包（序列化），通过网络发送到远程系统，由远程系统解包（反序列化），执行过程，并将结果返回给调用者。

实现技术

RPC 技术的实现有很多：

  • XML-RPC：使用 XML 编码调用，这是一种简单的协议，允许执行远程过程调用。
  • JSON-RPC：类似于 XML-RPC，但使用 JSON 而不是 XML 格式。
  • gRPC：由 Google 开发，使用 Protocol Buffers 作为接口定义语言和数据交换格式，支持多种编程语言。
  • SOAP：基于 XML 的协议，提供了丰富的数据类型和强大的兼容性，通常用于企业级应用。

RPC 广泛应用于各种分布式系统中，特别是使用微服务架构的应用程序。
